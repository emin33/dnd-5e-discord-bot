REFACTOR_PLAN.md targets the right things, but only Steps 0 and 1a have landed and the planned protocols.py/import-linter enforcement does not exist. ### Error handling & resilience — C

### Async & concurrency hygiene — C-

The codebase is saved from outright data corruption only by being single-event-loop, not by design discipline: newer code (tts_assembler's double-checked async locking, genuinely async cloud SDKs, executor-wrapped blocking IO in image_fal) is clean, while the core game loop has no coherent concurrency model — combat turn state is mutated by three unserialized paths, the one real lock globally serializes every session behind up-to-120s LLM calls, and at least three locks were declared for known races and never wired. A second theme is abandoned async subsystems left half-wired (fake streaming, dead seams). Rolled up: TurnLogger.flush does small sync file appends on the loop, _write_debug_log does sync IO when enabled, the lazy module singletons share an unguarded check-then-create pattern (benign on one loop, copied as a pattern), and bot close() neither ends sessions nor cancels in-flight immersion tasks.

**[P0/M] Combat turn state has three unserialized concurrent mutation paths and no lock anywhere in game/combat** (Main/dnd_bot/bot/views/combat_actions.py:226) — Combat state (turn order, HP) is mutated concurrently by (a) button views, (b) /combat slash commands (cogs/combat.py:512 next_turn, :1485/:1608 run_npc_turn), and (c) the orchestrator via on_message during COMBAT — only (c) holds the processing lock; the ONLY asyncio.Lock in Main/dnd_bot/game/ is session.py:167. Buttons have no interaction_check (zero hits in bot/), are never disabled before awaits, and have no done-guard: DashButton executes then calls on_action_complete (220-228), and game.py:578 calls on_turn_end() unconditionally even for failed results — a double-click yields two end_turn() calls and a skipped combatant. NPCTurnView disables its button only AFTER the multi-LLM-call turn completes (929-930), leaving the whole slow window re-clickable. — *Recommendation:* One owner for combat state: a per-channel asyncio.Lock (or turn-token check) inside CombatTurnCoordinator acquired by execute_action/end_turn/run_npc_turn; views disable items and stop() before any await; interaction_check restricts clicks to the acting player; on_turn_end fires only on success.

**[P1/L] Single global _processing_lock serializes ALL sessions because the singleton orchestrator carries per-request state** (Main/dnd_bot/game/session.py:605) — process_message wraps orchestrator work in one lock on the singleton manager purely because DMOrchestrator holds per-request context: session.py:608-609 set_session/set_scene_registry mutate orchestrator state that ~38 sites read during process_action. Same root as Architecture: "Per-session state lives on the process-wide orchestrator singleton" — this is the throughput/race face of it. — *Recommendation:* Per-session locks instead of global; pass session/scene_registry as process_action arguments (the plan's stated direction); move the pre-lock mutations inside the per-session critical section.

**[P1/M] Synchronous ChromaDB embedding/query calls on the event loop throughout the turn hot path** (Main/dnd_bot/memory/manager.py:637) — ChromaDB operations (in-process ONNX embedding + HNSW, sync) are called directly from async paths: manager.py:637 recall_for_context inside build_context every turn; orchestrator.py:1202/1348/1362 add_entity_description and :1227 add_narrative_chunk per turn inside process_action (also extending the global lock hold); session.py:233-242 loops add_entity_description over every KG entity at session start; game/knowledge/matcher.py:134 search_entities during matching. **[P1/M] Locks declared but never wired: Database._lock, _creation_states_lock, and bypassed _campaign_players_lock** (Main/dnd_bot/data/database.py:24) — Three locks exist as evidence of known races and protect nothing. database.py:24 _lock is never acquired while one shared aiosqlite connection serves all coroutines — interleaved execute/commit can commit each other's half-done transactions, and TransactionContext savepoints (189-206) can interleave release order. campaign.py:49 _creation_states_lock has zero `async with` usages (C35 in the 05-28 audit, still open). campaign.py:41 _campaign_players_lock is used only inside _add_campaign_player (52) which has NO callers — all five real mutation sites (96-97, 194-197, 283-286, 721-724, 999) duplicate the logic inline without the lock. — *Recommendation:* Wire Database._lock around execute/commit pairs (or give TransactionContext exclusive access); route all _campaign_players mutations through the locked helper; use or delete _creation_states_lock and the duplicate dict (character.py:41 holds a second independent copy).
